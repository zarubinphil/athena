# Банк вопросов грилла — interview-rubric (P1–P5)

> Драйвит полный грилл первого запуска (skill `setup-os`, Этап A). Generic, **без личных значений** — публикуем в репо.
>
> **Техника элиситации:** поведение-не-ярлык — спрашивай что агент должен УМЕТЬ в день 1, не «кто ты». Покрывай 6 мер AI-literacy: **Role / Task / Context / Process / Output / Verification**. JTBD-каркас для глубины: situation → motivation → progress → workaround → tradeoffs.
>
> **Поток фазы:** объясни 1 фразой зачем → AskUserQuestion (батч по домену) → **синтезируй** ответ (Карпаты: синтез-на-записи, не дамп сырья) → запиши в целевой файл → покажи → «дальше?».
>
> **Gate полноты:** каждый домен ниже — либо заполнен, либо явно пропущен-с-причиной (залогировать в `~/.claude/.athena-onboarding-progress`). Тихих дыр нет.

## Жёсткие правила грилла

- **Secrets:** собираем только КАРТУ (что есть / где лежит) → `secrets-map.md`. Значений НЕ спрашиваем и НЕ пишем в репо. Правило НИКОГДА.
- **Public ≠ private:** этот rubric и templates — generic, в репо. Сгенерённые owner.md/CLAUDE.md/манифесты — личные, в `~/.claude`/overlay/gitignored, НИКОГДА в публичный athena.
- **Instruction-budget:** глубина P4/P5 уходит в `references/*`, CLAUDE.md держит ≤200 строк указателями.
- **Чекпойнт:** после каждой фазы — стамп в `~/.claude/.athena-onboarding-progress`. Можно паузить и резюмить с последней фазы (это и есть «fast-path»).
