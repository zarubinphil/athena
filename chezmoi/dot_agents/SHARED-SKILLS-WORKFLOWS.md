# Shared Skills And Workflows

> Inventory regenerated by build_registry.py on bootstrap (Layer 2).

Applies to: Claude, Codex, and any agent that can read `~/.agents`

This is the shared source of truth for concrete skills, workflows, installation propagation, usage tracking, and GitHub activity signals.

## Required Read Rule

Before selecting or installing skills/workflows, every agent should read this file first, then read tool-specific instructions only when needed.

Read order:

1. `~/.agents/SHARED-SKILLS-WORKFLOWS.md`
2. `~/.agents/registry/CAPABILITY-PLANNING.md`
3. Relevant project `AGENTS.md`, `CLAUDE.md`, or `claud.md`
4. Selected skill/workflow body

If a project has stricter rules, follow the stricter project rule.

## Cross-Agent Propagation Rule

When a new skill, workflow, agent role, command, or related configuration is installed:

1. Install or mirror it for both Claude and Codex when technically applicable.
2. Register it in this file under `Shared Additions`.
3. Record expected locations for Claude and Codex.
4. Mark propagation status as one of:
   - `synced`: installed or mirrored for both Claude and Codex.
   - `codex-only`: only meaningful for Codex.
   - `claude-only`: only meaningful for Claude.
   - `pending`: not fully propagated yet.
5. Add a validation note explaining how stable operation was checked.
6. If the skill has a public source page or GitHub repository, add it to `~/.agents/INSTALLED-SKILL-SOURCES.md` and `~/.agents/TRACKED-INSTALLED-SOURCES.md`.

Do not delete or move existing skills during propagation unless the user explicitly approves a migration.

## Stable Operation Validation

For explicit concrete workflows, validate:

- The documented skill/agent/command exists at the declared path.
- Existing paths remain valid, or compatibility mirrors exist.
- The workflow has one canonical owner for each generated artifact.
- Agents read indexes/caches before raw large sources.
- No private data is copied across project boundaries by default.
- External actions such as push, publish, deploy, SSH, rsync, DNS, or credential changes require explicit user approval.

## Usage Tracking

Record usage in `~/.agents/SKILL-USAGE-LEDGER.md`.

Minimum fields:

`date | agent | skill_or_workflow | task_context | outcome | notes`

Usage frequency categories:

- `representative`: used often and internally convenient.
- `promising`: useful but needs more observations.
- `rare`: valuable for specialist work only.
- `archive-candidate`: duplicated, stale, or inconvenient.

When a skill/workflow is used for a meaningful task, append one row to the ledger. Do not log private case facts; describe task context generically.

## Installed Source Tracking

This system tracks the upstream pages and repositories of installed skills/tools.

Primary files:

- `~/.agents/INSTALLED-SKILL-SOURCES.md` — source URLs grouped by installed skills.
- `~/.agents/TRACKED-INSTALLED-SOURCES.md` — unique public source repositories/pages with latest stars when available.
- `~/.agents/GITHUB-STARS-LEDGER.md` — dated history of star counts.

Stars are a qualifying signal of external engagement or activity. They do not override local usefulness, security, correctness, or fit.

## Weekly Stars Update

Cadence: every 7 days.

Procedure:

1. Read `~/.agents/TRACKED-INSTALLED-SOURCES.md`.
2. For each valid public GitHub repository URL, fetch the current star count.
3. Append dated rows to `~/.agents/GITHUB-STARS-LEDGER.md`.
4. Update `~/.agents/TRACKED-INSTALLED-SOURCES.md` with the latest `stars`, `last_checked`, `status`, and notes.
5. If new installed sources appear in `~/.agents/.skill-lock.json`, add them to `~/.agents/INSTALLED-SKILL-SOURCES.md`.
6. Note repositories that are missing, private, deleted, or rate-limited.
7. Do not modify remote GitHub state.
